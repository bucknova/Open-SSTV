# Decoder algorithm

The SSTV decoder mirrors the well-known C reference [`slowrx`](https://github.com/windytan/slowrx)
(GPL). The build order below matches the milestones in the v1 plan and is
designed so each stage can be unit-tested in isolation against synthetic WAVs
generated by PySSTV.

## Pipeline

```
audio samples (int16 PCM)
        │
        ▼
to_mono_float32          (dsp_utils)
        │
        ▼
analytic_signal          (demod, scipy.signal.hilbert)
        │
        ▼
instantaneous_frequency  (demod, np.diff(np.unwrap(np.angle(...))) * fs / 2π)
        │
        ▼
detect_vis               (vis: 1900 Hz leader → 1200 Hz start bit
        │                     → 8 data bits + parity at 1100/1300 Hz
        │                     → stop bit; result is the mode VIS code)
        ▼
mode dispatch            (lookup in core.modes.MODE_TABLE)
        │
        ▼
find_line_starts         (sync: per-line 1200 Hz pulse + porch)
        │
        ▼
per-mode pixel decode    (decoder._decode_robot36 / _decode_martin /
        │                 _decode_scottie — each maps frequency to luma /
        │                 chroma per the mode's line layout)
        ▼
slant correction         (slant: re-time based on observed line spacing)
        │
        ▼
DecoderEvent stream      (VISDetected, LineDecoded, ImageComplete)
```

## Mode-specific notes

### Robot 36
- 320 × 240, 240 lines, ~150 ms per line.
- Even lines: Y component only. Odd lines: alternating R-Y / B-Y.
- Long sync pulse (9 ms) followed by a porch.
- The simplest mode to start with — pick this for the first end-to-end RX
  milestone.

### Martin M1
- 320 × 256, 256 lines, ~447 ms per line.
- G then B then R per line, separated by short separator pulses.
- Most popular European mode.

### Scottie S1
- 320 × 256, 256 lines, ~428 ms per line.
- G then B then R per line, but with the sync pulse **after** the green
  channel (the only common mode that does this — handle the off-by-one in
  ``find_line_starts`` carefully).
- Most popular US mode.

## Slant correction

A receiver clocked even ~100 ppm wrong produces a slanted image: every line
is sampled with a small cumulative offset. We measure the deviation between
observed line-start spacing and the mode-nominal spacing, derive a sample-rate
correction factor, and either resample the input via
``scipy.signal.resample_poly`` or warp the output image. Test by deliberately
resampling a fixture from 48000 Hz to 47900 Hz and asserting the corrected
image still passes the per-pixel error bound.

## Per-pixel error bounds (acceptance criteria)

For round-trip tests (PySSTV-encoded → our decoder → compare):

| SNR     | Luma MAE | Chroma MAE |
|---------|----------|------------|
| ∞       | < 5 %    | < 15 %     |
| 20 dB   | < 8 %    | < 20 %     |
| 10 dB   | < 15 %   | < 30 %     |
| 5 dB    | best-effort, no hard bound (just no crash) |
